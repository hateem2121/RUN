We are seeing frequent warnings (`SLOW QUERY`, `SLOW QUERY ALERT`, `SLOW REQUEST`, etc.) in our Replit-based platform (Node.js, NEON PostgreSQL, Drizzle ORM, etc.). The aim is to diagnose—not modify—the root causes by leveraging logs, plans, and performance metrics.  
**Do NOT propose fixes, code, or dependencies—focus on system observations.**

---

### Diagnostic Questionnaire

#### 1. General Observability
- What are the current slow query/request thresholds? Where are these set—logs, Drizzle ORM, NEON, or Express.js middleware?
- Which logs, dashboards, or tools (NEON dashboard, Drizzle logs, pg_stat_statements, etc.) show consistent performance metrics for these alerts?

#### 2. Query Patterns & Execution
- Are specific SQL queries, routes, or functions repeatedly flagged as slow?
- Do slow entries correlate with large result sets, missing indexes, non-batched operations, or N+1 patterns (see slow log, query plans)?
- For top slow queries, provide current EXPLAIN/EXPLAIN ANALYZE output and log sample.

#### 3. Database (NEON PostgreSQL)
- What is the cache hit ratio and are there signs of table/index bloat (NEON, pg_stat_statements stats)?
- Are there connection pool issues (e.g., maxed pool, idle timeouts) with the NEON serverless driver?  
- Are queries being batched when possible, or are frequent single-row ops occurring?
- Is the KV cache (@replit/database) helping during slow query windows, or are slow hits always uncached?

#### 4. Backend Server (Express.js)
- Do slow requests map to specific endpoints, middleware, or route handlers?
- Are there signs of resource starvation (CPU, RAM) or blocked event loop during slow periods?
- Is request volume or queue length abnormally high during SLOW REQUEST incidents?

#### 5. External Services
- Are there slow requests coinciding with calls to Nodemailer, @replit/object-storage, external HTTP APIs, etc.?
- Are timeouts or retries in these services contributing to the slow path?

#### 6. Client & Frontend
- Are SLOW REQUESTS visible to users as UI delays/timeouts, or only detectable in backend logs?
- Is TanStack Query (React Query) causing excess concurrent or stale refetches, impacting backend load?

#### 7. Temporal/Volume Patterns
- Are slow log spikes tied to user surges, scheduled jobs, or batch workloads?
- Are there patterns by time of day or per-user identity?

#### 8. Resource Limits/Quotas
- Are you approaching NEON or Replit project resource limits (RAM, vCPU, active connections), either at current or peak load?
- Any recent config or deployment changes preceding slow query/request alerts?

#### 9. Logs & Plans (Evidence Gathering)
- Collect slow query log entries (including duration, query, plan), noted periods, and any auto_explain or pg_stat_statements output.
- Note if auto_explain or log_min_duration_statement is enabled for PostgreSQL, and typical durations flagged.

---

**Instructions:**  
Please answer each item based on your logs, dashboards, or observed system metrics.
